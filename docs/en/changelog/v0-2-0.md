# v0.2.0 — ITool Architecture Rewrite & Runtime Bridge

**Release Date**: 2026-06-12

Migrated from monolithic CommandFn function pointer architecture to object-oriented ITool interface + X-macro registration system, adding runtime bridge, debugger integration, ClassDB documentation queries, and 149+ tools.

### Breaking Changes

- **Tool registration mechanism rewrite** — migrated from `CommandFn` function pointers + codegen script to `ITool` interface + X-macro auto-registration
  - Removed `tools/codegen.py` and `// @tool register` annotation mechanism
  - Each tool is now a standalone `.hpp` file implementing the `ITool` interface (`name()`, `category()`, `execute_impl()`, etc.)
  - Auto-collected via `register/*.hpp` X-macro files, no `.cpp` files needed
- **Unified tool naming** — all tool names migrated to clean `snake_case`
  - `get_node_position` → removed (replaced by generic `get_node_property` fallback)
  - `create_gdscript` → `write_gd_script`, `edit_gdscript` → `patch_gd_script`, etc.
  - Old monolithic files (`node.cpp`, `property.cpp`, `scene.cpp`, etc.) split into individual tools
- **Category system restructured** — from 16 flat command groups to two-level category tree
  - `editor_tools/scene_tree`, `editor_tools/scripts`, `editor_tools/workspace`, `runtime_tools/bridge`, etc.
  - Auto-discovery + `find_tool` full-text search engine
- **YAML database downgraded to documentation-only** — `node_props/db/*.yaml` (283) and `node_resource/db/*.yaml` (419) no longer participate in build/registration
  - Property tools now query Godot ClassDB at runtime
- **Polling mechanism change** — from `_on_process_frame` signal to `_process()` virtual function override

### New Features

- **ITool framework** — `tool_base.cpp/.hpp` provides full type validation, schema generation, category management
  - `IToolAdapter` bridges GDScript SDK tools into the ITool system
  - `HandlerRegistry` dual-track registration: ITool primary table + CommandFn sidetable
- **Runtime bridge system** — bidirectional communication between editor and game processes
  - `GameBridgeNode` (TCP server in game process, port 9601)
  - `RuntimeBridge` (TCP client on editor side)
  - 6 bridge tools: scene tree query, property read/write, method calls, screenshots, input simulation
  - Automatic game start/stop detection via `is_playing_scene()`
- **Runtime lifecycle tools** (6) — `run_project`, `run_current_scene`, `run_specific_scene`, `stop_project`, `pause_project`, `set_movie_maker`
- **Debugger integration** (15 tools) — FPS, memory, performance monitors, breakpoint management, step debugging, stack traces, locals
- **ClassDB documentation query tools** (5) — `get_class_list`, `get_inheritance_chain`, `get_property_doc`, `get_method_doc`, `get_enum_doc`
- **Workspace tools** (5) — 2D/3D/Script/AssetLib workspace switching
- **Console tools** (4) — output retrieval, error/warning filtering, clear
- **Animation tools** (5) — AnimationPlayer creation, animation clips, tracks, keyframes
- **UI control tools** (4) — control creation, StyleBox, layout presets, theme overrides
- **Shader tools** (3) — create, read, apply preset
- **TileMap tools** (3) — info retrieval, cell set/erase
- **Export tools** (2) — preset listing, project export
- **Project scaffolding** (1) — `create_project` for new projects
- **Project graph visualization** (1) — `get_project_graph` dependency graph
- **Viewport capture** — `capture_viewport`, `capture_game_viewport`
- **Signal management** (4) — connect/disconnect signals, list signals, get connections
- **Node groups** (4) — add/remove from groups, query group members
- **Resource operations** (6) — save/load/create/duplicate/clear/query resources
- **Property fallback tools** (2) — generic `get_node_property`/`set_node_property` covering all node properties
- **C++ YAML test engine** — `TestEngine` executes YAML tests in-editor
- **Build optimizations** — sccache/ccache auto-detect, Unity build, lld-link, ryml header-only YAML

### Refactoring

- **Four-layer tool system** (ADR-015) — Layer 0 fallback → Layer 1 specialized → Layer 2 editor → Layer 3 docs
- **SDK integrated via IToolAdapter** — GDScript SDK tools bridged into ITool system
- **Tool category description system** — `category_description()` auto-beautifies category labels
- **Flattened directory structure** — `extensions/src/built_in/tools/` organized by functional domain
- **HTTP server modularized** — Origin validation security hardening, reentrancy prevention (`save_scene` crash fix)
- **Build system** — removed PCH, `build.py` cache auto-detect + SSL fallback retry

### Fixes

- `save_scene` crash: `EditorProgress::step()` → `Main::iteration()` → reentrancy → switched to `save_scene_as(path, false)` to skip preview
- `delete_file`/`paste_node`/`run_scene` semantic error fixes
- HTTP Origin validation vulnerability fix
- `HashMap` range-for `erase()` iterator invalidation → UB crash
- Integer type validation now accepts float (`tool_base.cpp`)
- `add_node` now accepts `name` parameter alias
- `handler_registry` category label underscore handling

### Removed

- `tools/codegen.py` and `// @tool register` codegen mechanism
- Old monolithic tool files (`cmd_node.cpp`, `cmd_scene.cpp`, `cmd_property.cpp`, etc.)
- `tools/tool_schemas.json` (tool schemas now dynamically generated via ITool interface)
