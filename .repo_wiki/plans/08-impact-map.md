### 1.2 WHY — 业务目标

| 维度 | 目标 | 衡量标准 |
|------|------|---------|
| **技术领先** | 成为首个原生 GDExtension MCP 工具 | 架构对比文档覆盖 6 个竞品 |
| **安全可靠** | AI 操作零数据损坏 | 破坏性操作 100% 有确认机制 |
| **开发者体验** | 从安装到首次使用 <5 分钟 | AssetLib 一键安装 + 首次 MCP 调用 |
| **闭环能力** | 编辑→运行→检查→修复全链路 | 运行时桥接 + 操作回滚 |

| 行为者 | 角色 | 核心需求 | 影响路径 |
|--------|------|---------|---------|

| **游戏开发者** | 使用 Godot 进行游戏开发 | 简单部署、多实例测试 | HOW: 简单部署 → WHAT: L1 AssetLib |

| **项目维护者** | 维护 GodotMCP 项目 | 可测试、可发布 | HOW: CI/CD → WHAT: L1 发布流水线 |

## 2. SMART 目标

### 2.1 项目级 SMART 目标

> **在 27.5 个工作日内，完成 GodotMCP 从 v0.2.2-dev1 到 v1.0.0 的架构演进，实现 MCP 标准兼容（全量/渐进双模式）、AI 操作安全防护（确认+回滚）、WebSocket 同端口多实例桥接，以及 AssetLib 自动化发布，覆盖 Windows/Linux/macOS 三平台。**

| SMART 维度 | 验证 |
|-----------|------|
| **S**pecific | 4 个 Sprint、11 个任务节点、明确的交付物清单 |
| **M**easurable | 155+ 工具、100% 破坏性操作确认率、WebSocket 帧 roundtrip 测试通过 |
| **A**chievable | 基于 Godot 源码验证的 API 可用性，无技术阻断点 |
| **R**elevant | 直接对齐"安全、高效、闭环"三大业务目标 |
| **T**ime-bound | 27.5 工作日，3 个 Sprint，明确的里程碑日期 |

### 2.2 Sprint 级 SMART 目标

#### Sprint 1: 安全基线 + 兼容性 + 协议修复

| 维度 | 目标 |
|------|------|
| **S** | 实现渐进式披露开关、破坏性操作确认 UI、桥接协议修复、端口回退 |
| **M** | `tools/list` 双模式切换；14 个破坏性工具有确认弹窗；桥接消息零错位 |
| **A** | 4 个独立子任务，并行执行，文件域隔离 |
| **R** | 安全防护（Q2）+ 标准兼容（Q1）+ 闭环调试基础（Q3） |
| **T** | 3 工作日（并行执行后关键路径为 Q2 的 3d） |

**验收标准**：
- [ ] `ProjectSettings godot_mcp/progressive_disclosure = false` → `tools/list` 返回 153 工具

- [ ] 桥接连接包含 `\n` 的 JSON 消息 → 正确解析无错位
- [ ] 第二个编辑器实例启动 → 自动回退到 `:9601`，Dock 显示警告

#### Sprint 2: 透明度 + 自修复 + 发布

| **M** | 搜索过滤延迟 <100ms；回滚精确匹配 operation_id；CI 三平台构建成功 |

| **R** | 高效发现（M3'）+ 安全防护（L2）+ 简单部署（L1） |
| **T** | 4 工作日 |

**验收标准**：
- [ ] MCP Console 搜索栏输入 "delete" → 仅显示 delete 相关工具调用
- [ ] MCP Console 每行有 Revert 按钮（仅 supports_undo 条目）
- [ ] `rollback_operation(operation_id="op_xxx")` → 精确撤销对应操作

#### Sprint 3: 增强桥接

|------|------|

## 3. 项目管理铁三角

### 3.1 优先级定调

```
         质量 (Quality)
            ╱╲
           ╱  ╲
          ╱    ╲
         ╱  ★   ╲        ★ = 当前优先级
        ╱        ╲
       ╱──────────╲
  时间            成本
 (Time)          (Cost)
```

**优先级排序**：

| 优先级 | 维度 | 约束 | 说明 |
|--------|------|------|------|
| **P0** | **质量** | 零数据损坏、100% 破坏性确认、协议兼容 | 安全与正确性不可妥协 |
| **P1** | **时间** | 27.5 工作日，3 Sprint | 明确的里程碑交付 |
| **P2** | **成本** | 1 人全职 + Subagent 并发 | 人力固定，通过并行优化时间 |

**决策原则**：
- 当质量与时间冲突时：**保质量**，时间可延长 1 Sprint（+10d 缓冲）
- 当时间与成本冲突时：**保时间**，增加 Subagent 并发度
- 当质量与成本冲突时：**保质量**，成本不可削减测试

### 3.2 范围管理

| 范围类型 | 内容 | 决策 |
|---------|------|------|
| **必须做 (Must)** | Q1, Q2, Q3, L3 | Sprint 1 交付，阻断后续 |
| **应该做 (Should)** | M3', L2, L1 | Sprint 2 交付，核心价值 |
| **可以做 (Could)** | L4+L5 | Sprint 3 交付，差异化能力 |
| **不做 (Won't)** | M1, M2 | 记录技术债务，未来评估 |

### 3.3 管理储备与应急储备

| 储备类型 | 大小 | 用途 | 触发条件 |
|---------|------|------|---------|
| **应急储备** (Contingency) | 5d（每个 Sprint 的 15-20%） | 已识别风险的应对 | R1-R10 中任何风险触发 |
| **管理储备** (Management) | 5d（项目级） | 未识别风险的应对 | 范围变更、技术阻断 |
| **总缓冲** | 10d | | |
| **总工期（含储备）** | 27.5d + 10d = 37.5d | | |

**储备分配**：

| Sprint | 基础工期 | 应急储备 | 含储备工期 |
|--------|---------|---------|----------|

| Sprint 3 | 9d | 2.5d | 11.5d |
| 管理储备 | — | 5d | 5d |
